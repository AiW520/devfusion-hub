# Docker+Kubernetes 方案一：微服务部署平台

## 📚 项目介绍

### 1.1 项目概述

这是一个完整的云原生微服务部署平台，展示了如何使用 Docker 容器化 Go 微服务，并使用 Kubernetes 进行编排、扩缩容、服务发现。项目是面试云原生岗位的高分项目！

**核心功能：**
- Docker 镜像构建与优化
- 多架构镜像支持
- Kubernetes 部署配置
- 服务发现与配置管理
- 自动扩缩容（HPA）
- 金丝雀发布与回滚
- 滚动更新策略
- 日志收集与监控

### 1.2 核心技术亮点

```
云原生架构：
├── Docker                    # 容器化
│   ├── 多阶段构建           # 减小镜像体积
│   ├── 镜像优化             # distroless/alpine
│   └── 安全扫描             # Trivy
├── Kubernetes                # 编排
│   ├── Deployment            # 应用部署
│   ├── Service               # 服务发现
│   ├── ConfigMap/Secret      # 配置管理
│   ├── HPA                   # 自动扩缩容
│   ├── Ingress               # 入口网关
│   └── PDB                   # PodDisruptionBudget
├── Helm                      # 包管理
├── Prometheus + Grafana      # 监控
├── Jaeger                    # 链路追踪
└── ArgoCD                    # GitOps 部署
```

### 2.2 Docker 配置

#### 2.2.1 Dockerfile.multi - 多阶段构建

```dockerfile
# =============================================================================
# 第一阶段：构建阶段（Builder）
# =============================================================================
# 使用 Go 官方镜像作为构建环境
# alpine 是精简版，只包含必要的工具
# AS builder 给阶段起别名，后面引用时用
FROM --platform=$BUILDPLATFORM golang:1.21-alpine AS builder

# 设置工作目录
WORKDIR /build

# =============================================================================
# 安装构建依赖
# =============================================================================
# 安装 Git（go mod 需要）
# 安装构建工具
RUN apk add --no-cache git make bash

# =============================================================================
# 设置 Go 编译参数
# =============================================================================
# TARGETOS 和 TARGETARCH 由 docker buildx 自动传入
# CGO_ENABLED=0 表示静态编译，不依赖 C 库
# GOOS 和 GOARCH 适配目标平台
ARG TARGETOS
ARG TARGETARCH

# 设置 GOPROXY，加快依赖下载
ENV GOPROXY=https://goproxy.cn,direct

# =============================================================================
# 复制源代码
# =============================================================================
# 先复制 go.mod 和 go.sum（如果有）
# 这样可以利用 Docker 缓存层，依赖只在变更时重新下载
COPY go.mod go.sum ./

# 下载依赖（单独执行，可以缓存）
RUN go mod download

# 复制源代码
COPY . .

# =============================================================================
# 编译二进制文件
# =============================================================================
# 交叉编译：支持 linux/amd64 和 linux/arm64
RUN CGO_ENABLED=0 GOOS=${TARGETOS} GOARCH=${TARGETARCH} \
    go build -ldflags="\
    -s -w \
    -X main.Version=${VERSION:-v1.0.0} \
    -X main.BuildTime=$(date -u '+%Y-%m-%d %H:%M:%S') \
    " \
    -o user-service \
    ./cmd/server/main.go

# =============================================================================
# 第二阶段：运行阶段（Runtime）
# =============================================================================
# 使用 distroless 镜像，体积最小（约 20MB）
# distroless 特点：
# - 只包含运行时需要的库
# - 没有 shell、package manager
# - 更安全，适合生产环境
#
# 可选镜像：
# - gcr.io/distroless/static:debian12  # 最精简
# - alpine:latest                      # 小但有 shell（约 7MB）
# - debian:stable-slim                # 较大但完整（约 80MB）
FROM --platform=$TARGETPLATFORM gcr.io/distroless/static:debian12 AS runtime

# 添加非 root 用户
# Kubernetes 最佳实践：以非 root 用户运行容器
# distroless 镜像自带 nonroot 用户，UID 65532
USER nonroot:nonroot

# 设置工作目录
WORKDIR /app

# =============================================================================
# 从构建阶段复制二进制文件
# =============================================================================
# --from=builder 引用第一阶段的产物
COPY --from=builder /build/user-service .

# 复制配置文件（如果需要）
# COPY --from=builder /build/config /app/config

# =============================================================================
# 暴露端口
# =============================================================================
# 这个注释帮助其他人了解服务端口
# 实际使用 CONTAINER_PORT 环境变量覆盖
EXPOSE 8080

# =============================================================================
# 健康检查
# =============================================================================
# Kubernetes 使用 readnessProbe 和 livenessProbe
# 这里定义的是 Docker 健康检查（兼容性考虑）
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD ["wget", "-q", "--spider", "http://localhost:8080/healthz"]

# =============================================================================
# 启动命令
# =============================================================================
# 使用数组格式，这是 shellcheck 推荐的方式
# 避免 shell 解析问题
ENTRYPOINT ["./user-service"]

# =============================================================================
# 多平台构建说明
# =============================================================================
# 构建命令：
# docker buildx build \
#   --platform linux/amd64,linux/arm64 \
#   --tag your-registry/user-service:v1.0.0 \
#   --push \
#   .
#
# --platform: 指定目标平台
# --tag: 镜像标签
# --push: 推送到镜像仓库
```

#### 2.2.2 Dockerfile - 简单版本

```dockerfile
# 如果只需要单平台构建，可以使用这个简化版本
FROM golang:1.21-alpine AS builder

WORKDIR /build

# 设置 GOPROXY
ENV GOPROXY=https://goproxy.cn,direct

# 复制源码
COPY . .

# 编译
RUN CGO_ENABLED=0 go build -o user-service ./cmd/server/main.go

# 运行阶段
FROM alpine:latest

WORKDIR /app

# 安装 CA 证书（某些 HTTPS 请求需要）

# 从构建阶段复制

RUN adduser -D -u 1000 appuser
USER appuser

EXPOSE 8080

ENTRYPOINT ["./user-service"]
```

### 2.3 Kubernetes 配置

#### 2.3.1 deployment.yaml - 应用部署

```yaml
# =============================================================================
# Deployment 配置
# 定义应用程序的部署方式
# =============================================================================

apiVersion: apps/v1                                      # API 版本
kind: Deployment                                         # 资源类型

metadata:
  name: user-service                                     # Deployment 名称
  namespace: production                                  # 命名空间
  labels:
    app: user-service                                    # 应用标签
    version: v1.0.0                                      # 版本标签
  annotations:
    # 描述信息
    description: "User Service - 用户服务"
    # 变更日志
    kubernetes.io/change-cause: "Update to v1.0.0"

spec:
  # =============================================================================
  # 副本数控制
  # =============================================================================
  replicas: 3                                            # 期望副本数

  # =============================================================================
  # 滚动更新策略
  # =============================================================================
  strategy:
    type: RollingUpdate                                 # 滚动更新（还有 Recreate）
    rollingUpdate:
      maxSurge: 1                                       # 最大超出副本数
      # 更新时最多可以创建 1 个新 Pod
      # 配合 replicas=3，更新时最多有 4 个 Pod

      maxUnavailable: 0                                # 最大不可用数
      # 设为 0 保证更新期间服务不中断
      # 配合 maxSurge=1，保证至少 3 个可用 Pod

  # =============================================================================
  # 选择器
  # =============================================================================
  selector:
    matchLabels:
      app: user-service                                 # 匹配此标签的 Pod

  # =============================================================================
  # Pod 模板
  # =============================================================================
  template:
    metadata:
      labels:
        app: user-service
        version: v1.0.0
      annotations:
        # Prometheus 监控注解
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/metrics"

    spec:
      # =============================================================================
      # Pod 配置
      # =============================================================================

      # 终止宽限期
      # Pod 被删除后，容器的宽限期
      terminationGracePeriodSeconds: 30                 # 默认 30 秒

      # 安全上下文
      securityContext:
        runAsNonRoot: true                             # 必须以非 root 运行
        runAsUser: 65532                                # nonroot 用户的 UID
        fsGroup: 65532                                  # 文件系统组
        runAsGroup: 65532

      # 服务账号
      serviceAccountName: user-service                 # 使用的 ServiceAccount

      # 亲和性配置
      affinity:
        # Pod 反亲和性：避免同一应用实例分布在同一节点
        # 保证高可用，避免单点故障
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchExpressions:
                    - key: app
                      operator: In
                      values:
                        - user-service
                topologyKey: kubernetes.io/hostname

        # 节点亲和性：可以选择部署到特定节点
        # nodeAffinity:
        #   preferredDuringSchedulingIgnoredDuringExecution:
        #     - weight: 50
        #       preference:
        #         matchExpressions:
        #           - key: node-type
        #             operator: In
        #             values:
        #               - compute-optimized

      # =============================================================================
      # 容器配置
      # =============================================================================
      containers:
        - name: user-service                           # 容器名称
          image: your-registry/user-service:v1.0.0      # 镜像地址
          imagePullPolicy: IfNotPresent                 # 镜像拉取策略

          # 端口配置
          ports:
            - name: http                                # 端口名称
              containerPort: 8080                      # 容器监听端口
              protocol: TCP

            - name: grpc                               # gRPC 端口
              containerPort: 9090
              protocol: TCP

          # =============================================================================
          # 资源管理
          # =============================================================================
          resources:
            requests:
              cpu: "100m"                               # 最小 CPU（0.1 核）
              memory: "128Mi"                          # 最小内存
            limits:
              cpu: "500m"                               # 最大 CPU（0.5 核）
              memory: "512Mi"                          # 最大内存

          # =============================================================================
          # 环境变量
          # =============================================================================
          env:
            # 服务端口（从 ConfigMap 引用）
            - name: APP_PORT
              valueFrom:
                configMapKeyRef:
                  name: user-service-config
                  key: app.port
                  optional: false                      # 必须存在

            # 数据库密码（从 Secret 引用）
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: user-service-secrets
                  key: db-password
                  optional: false

            # 时区设置
            - name: TZ
              value: "Asia/Shanghai"

          # =============================================================================
          # 健康检查
          # =============================================================================

          # 存活探针：检查进程是否存活
          # 如果检查失败，Kubernetes 会重启容器
          livenessProbe:
            httpGet:
              path: /healthz
              port: http
            initialDelaySeconds: 10                    # 启动后 10 秒开始检查
            periodSeconds: 10                           # 每 10 秒检查一次
            timeoutSeconds: 3                           # 超时 3 秒视为失败
            failureThreshold: 3                         # 连续 3 次失败重启容器

          # 就绪探针：检查是否接收流量
          # 如果检查失败，K8s 会从 Service 中移除该 Pod
          # 避免将流量发送到未就绪的实例
          readinessProbe:
            httpGet:
              path: /readyz
              port: http
            initialDelaySeconds: 5                     # 启动后 5 秒开始检查
            periodSeconds: 5                            # 每 5 秒检查一次
            timeoutSeconds: 2                           # 超时 2 秒视为失败
            successThreshold: 1                         # 成功 1 次即认为就绪
            failureThreshold: 3                         # 连续 3 次失败标记为未就绪

          # 启动探针：检查应用启动是否完成
          # 在启动期间，liveness 和 readiness 探针会被禁用
          # 适用于启动时间较长的应用
          startupProbe:
            httpGet:
              path: /healthz
              port: http
            initialDelaySeconds: 0
            periodSeconds: 5
            timeoutSeconds: 3
            failureThreshold: 30                        # 最多等待 150 秒启动完成
            # 30 * 5 = 150 秒

          # =============================================================================
          # 卷挂载
          # =============================================================================
          volumeMounts:
            - name: tmp                                 # 临时目录
              mountPath: /tmp

            - name: cache                               # 缓存目录
              mountPath: /app/cache

          # =============================================================================
          # 安全上下文
          # =============================================================================
          securityContext:
            allowPrivilegeEscalation: false            # 禁止提权
            readOnlyRootFilesystem: true               # 只读根文件系统
            capabilities:
              drop:
                - ALL                                  # 移除所有 capabilities

      # =============================================================================
      # 初始化容器
      # =============================================================================
      # 在主容器启动前运行，完成数据初始化、配置加载等
      initContainers:
        - name: wait-for-db
          image: busybox:1.36
          command:
            - sh
            - -c
            - |
              echo "Waiting for database..."
              # 等待数据库就绪
              until nc -z ${DB_HOST:-postgres} ${DB_PORT:-5432}; do
                echo "Database not ready, waiting..."
                sleep 2
              done
              echo "Database is ready!"
          env:
            - name: DB_HOST
              valueFrom:
                configMapKeyRef:
                  name: user-service-config
                  key: db.host
            - name: DB_PORT
              valueFrom:
                configMapKeyRef:
                  name: user-service-config
                  key: db.port

      # =============================================================================
      # 卷配置
      # =============================================================================
      volumes:
        - name: tmp
          emptyDir: {}                                  # 临时目录

        - name: cache
          emptyDir:
            medium: Memory                             # 内存文件系统
            sizeLimit: 256Mi                           # 限制大小
```

#### 2.3.2 service.yaml - 服务发现

```yaml
# =============================================================================
# Service 配置
# 提供 Pod 的固定访问入口（ClusterIP）
# 自动负载均衡和 DNS 解析
# =============================================================================

apiVersion: v1
kind: Service
metadata:
  name: user-service                                   # Service 名称
  namespace: production
  labels:
    app: user-service

  annotations:
    # 描述
    description: "User Service Internal Endpoint"

spec:
  # =============================================================================
  # 服务类型
  # =============================================================================
  type: ClusterIP                                      # 集群内部访问

  # =============================================================================
  # 端口配置
  # =============================================================================
  ports:
    # HTTP 端口
    - name: http
      port: 80                                        # Service 端口
      targetPort: http                                 # 指向容器的命名端口
      protocol: TCP

    # gRPC 端口
    - name: grpc
      port: 90
      targetPort: grpc
      protocol: TCP

  # =============================================================================
  # 选择器
  # =============================================================================
  selector:
    app: user-service                                  # 匹配 Deployment 的 Pod

  # =============================================================================
  # 会话亲和性（可选）
  # =============================================================================
  sessionAffinity: None                                # None: 无会话保持
  # sessionAffinity: ClientIP                          # ClientIP: 同一 IP 转发到同一 Pod

  # =============================================================================
  # 外部流量策略
  # =============================================================================
  # externalTrafficPolicy: Cluster                     # Cluster: 流量可能转发到其他节点
  # externalTrafficPolicy: Local                        # Local: 只转发到本节点 Pod

---

# =============================================================================
# Headless Service（可选）
# 用于 StatefulSet，Pod 有固定 DNS
# =============================================================================

apiVersion: v1
kind: Service
metadata:
  name: user-service-headless
  namespace: production
spec:
  type: ClusterIP
  clusterIP: None                                     # Headless 的关键：clusterIP 为 None
  ports:
    - name: http
      port: 80
      targetPort: http
  selector:
    app: user-service
```

    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"

    # 代理缓冲
    nginx.ingress.kubernetes.io/proxy-buffering: "on"

# 根据指标自动调整 Pod 副本数
# =============================================================================

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: user-service-hpa

  # =============================================================================
  # 目标 Deployment

  # =============================================================================
  # 扩缩容指标
  # =============================================================================
  metrics:
    # CPU 使用率
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70                       # 平均 CPU 70% 时扩容

    # 内存使用率
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80                      # 平均内存 80% 时扩容

    # 自定义指标（需要 Prometheus Adapter）
    # - type: Pods
    #   pods:
    #     metric:
    #       name: http_requests_per_second
    #     target:
    #       type: AverageValue
    #       averageValue: "1000"

  # =============================================================================
  # 扩缩容行为
  # =============================================================================
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300                  # 缩容冷却时间（5分钟）
      policies:
        # 每次最多缩容 2 个 Pod
        - type: Percent
          value: 10
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0                    # 扩容无需冷却
      policies:
        # 每 15 秒最多扩容 4 个 Pod
        - type: Percent
          value: 100
          periodSeconds: 15
        - type: Pods
          value: 4
          periodSeconds: 15
      selectPolicy: Max                                # 取最大值

## 📝 代码关键点说明

### 3.1 Docker 多阶段构建原理

```
构建阶段分离的好处：
1. 减小镜像体积
   - 构建阶段需要 Go 编译器和源码
   - 运行阶段只需要编译后的二进制文件
   
2. 提高安全性
   - 运行镜像不包含编译器
   - 没有 shell（distroless）

3. 多平台支持
   - 可以同时构建 amd64 和 arm64 镜像
   - 一套配置支持多种架构
```

### 3.2 Kubernetes 探针配置

```
探针类型对比：
┌─────────────┬──────────────┬─────────────────┬──────────────┐
│   探针      │    失败后果   │    使用场景      │   失败阈值   │
├─────────────┼──────────────┼─────────────────┼──────────────┤
│ liveness    │ 重启容器      │ 检查进程存活     │ 3次          │
│ readiness   │ 移除流量     │ 检查就绪状态     │ 3次          │
│ startup     │ 禁用其他探针  │ 启动时间长的应用  │ 30次(150秒) │
└─────────────┴──────────────┴─────────────────┴──────────────┘

最佳实践：
- liveness: initialDelay > 启动时间
- readiness: initialDelay < 启动时间
- startup: failureThreshold * period > 启动时间
```

### 3.3 HPA 扩缩容原理

```
扩缩容流程：
1. HPA Controller 定期检查指标
2. 计算期望副本数：desiredReplicas = ceil(sum(currentMetrics) / target)
3. 与当前副本数比较
4. 更新 Deployment 副本数
5. Deployment Controller 创建/删除 Pod

冷却时间：
- scaleDown.stabilizationWindowSeconds: 缩容冷却
- 避免抖动：流量突增后立即缩容又扩容
```

## 🎯 面试要点

### 问题 1：Docker 和 Kubernetes 的区别是什么？

**参考答案：**
```
Docker（容器运行时）：
- 打包应用及其依赖
- 容器化单个应用
- 本地开发和部署

Kubernetes（容器编排平台）：
- 管理多个容器
- 自动调度、扩缩容
- 服务发现、负载均衡
- 自我修复

关系：
- K8s 使用 Docker（或 containerd）作为容器运行时
- Docker 提供容器化能力
- K8s 提供编排能力
```

### 问题 2：K8s 中 Pod 的生命周期是什么？

**参考答案：**
```
Pod 生命周期：
1. Pending：Pod 被调度
2. ContainerCreating：创建容器镜像
3. Running：容器运行中
4. Succeeded：正常退出（一次性任务）
5. Failed：异常退出
6. Unknown：节点失联

重启策略：
- Always：总是重启（Deployment 默认）
- OnFailure：失败时重启
- Never：从不重启

状态探针：
- livenessProbe: 存活检查，失败重启
- readinessProbe: 就绪检查，失败移除流量
- startupProbe: 启动检查，期间禁用其他探针
```

### 问题 3：K8s 中 Service 如何实现负载均衡？

**参考答案：**
```
Service 类型：
- ClusterIP：集群内部访问
- NodePort：节点端口外部访问
- LoadBalancer：云厂商负载均衡器
- ExternalName：DNS CNAME

负载均衡原理：
1. kube-proxy 监听 Service 和 Endpoint 变化
2. 为每个 Service 创建 iptables/ipvs 规则
3. 流量通过规则转发到 Pod
4. 随机或轮询选择目标 Pod

新方式（Kubernetes 1.11+）：
- 使用 IPVS 模式
- 支持更多负载均衡算法
- 性能更好
```

### 问题 4：如何实现零宕机部署？

**参考答案：**
```
零宕机部署关键点：
1. 滚动更新策略
   maxSurge: 1      # 额外 1 个 Pod
   maxUnavailable: 0 # 不允许不可用

2. 就绪探针
   - 确保新 Pod 就绪后才接收流量
   - 新旧 Pod 并存期间服务不中断

3. PDB（PodDisruptionBudget）
   - 维护期间保证最小可用数
   - 防止所有 Pod 同时被驱逐

4. 优雅关闭
   - preStop hook 等待流量排出
   - terminationGracePeriodSeconds 足够长

5. 健康检查
   - readinessProbe 准确反映应用状态
```

### 问题 5：Go GMP 调度模型是什么？

**参考答案：**
```
GMP 模型：
- G（Goroutine）：协程，轻量级线程
- M（Machine）：系统线程
- P（Processor）：逻辑处理器

工作流程：
1. P 从本地队列获取 G
2. G 在 M 上执行
3. G 执行系统调用时，P 分离 M
4. 空闲 P 偷取其他 P 的 G

调度策略：
- 工作窃取：空闲 P 从其他 P 偷 G
- 公平调度：G 执行一定时间后让出
- 系统调用：G 阻塞时不阻塞 M
```
