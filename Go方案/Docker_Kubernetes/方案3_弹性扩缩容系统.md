# Docker+Kubernetes 方案三：弹性扩缩容系统

## 📚 项目介绍

### 1.1 项目概述

这是一个完整的弹性扩缩容系统，展示了 Kubernetes HPA/VPA/KEDA 的高级用法、自定义指标采集、Prometheus 监控、告警处理等核心能力。项目是展示云原生弹性设计和生产级高可用架构的面试高分项目！

**核心功能：**
- 基于 CPU/内存的自动扩缩容
- 基于自定义指标（QPS、延迟、队列长度）的扩缩容
- 基于事件的扩缩容（Kafka 消息队列、Redis 队列）
- CronJob 定时扩缩容
- 渐进式扩缩容（避免震荡）
- 缩容冷却保护
- 缩容到零（成本优化）

### 1.2 核心技术亮点

```
弹性扩缩容架构：
├── Kubernetes HPA                  # Horizontal Pod Autoscaler
│   ├── 基于资源指标               # CPU/内存
│   ├── 基于自定义指标             # Prometheus 指标
│   └── 基于外部指标               # 队列深度
├── KEDA                            # Kubernetes Event-driven Autoscaling
│   ├── Kafka 触发器               # 队列消息数
│   ├── Redis 触发器              # List 长度
│   ├── RabbitMQ 触发器            # 队列长度
│   └── Cron 触发器               # 定时扩缩容
├── VPA                            # Vertical Pod Autoscaler
│   ├── 自动推荐资源              # 基于历史使用
│   └── 自动更新资源限制          # 推荐值应用
├── Prometheus                      # 指标采集
│   ├── 自定义指标                 # 业务指标
│   └── AlertManager              # 告警
└── 监控和可视化
    ├── Grafana                    # 可视化
    └── Kiali                      # 服务网格可视化
```

### 2.1 Prometheus 自定义指标

#### 2.1.1 Go 应用暴露指标

```go
package main

import (
	"net/http"
	"strconv"
	"time"

	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promhttp"
)

/*
Prometheus 指标类型：
1. Counter - 计数器，只增不减
2. Gauge - 仪表，可增可减
3. Histogram - 直方图，统计分布
4. Summary - 摘要，统计分位数
*/

var (
	// =============================================================================
	// HTTP 请求指标
	# =============================================================================
	
	// 请求计数器
	httpRequestsTotal = prometheus.NewCounterVec(
		prometheus.CounterOpts{
			Name: "http_requests_total",
			Help: "Total number of HTTP requests",
		},
		[]string{"method", "endpoint", "status"}, // 标签
	)

	// 请求延迟直方图
	httpRequestDuration = prometheus.NewHistogramVec(
		prometheus.HistogramOpts{
			Name:    "http_request_duration_seconds",
			Help:    "HTTP request latency in seconds",
			// 分桶：0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10
			Buckets: prometheus.ExponentialBuckets(0.005, 2, 10),
		},
		[]string{"method", "endpoint"},
	)

	// 正在处理的请求数
	httpRequestsInFlight = prometheus.NewGauge(
		prometheus.GaugeOpts{
			Name: "http_requests_in_flight",
			Help: "Number of HTTP requests currently being processed",
		},
	)

	// =============================================================================
	# 业务指标
	# =============================================================================

	// 订单处理计数
	ordersProcessed = prometheus.NewCounterVec(
		prometheus.CounterOpts{
			Name: "orders_processed_total",
			Help: "Total number of orders processed",
		},
		[]string{"status"}, // success, failed
	)

	// 订单处理延迟
	orderProcessingDuration = prometheus.NewHistogramVec(
		prometheus.HistogramOpts{
			Name:    "order_processing_duration_seconds",
			Help:    "Order processing duration in seconds",
			Buckets: []float64{0.1, 0.5, 1, 2, 5, 10, 30},
		},
		[]string{"order_type"},
	)

	// 活跃订单数
	activeOrders = prometheus.NewGauge(
		prometheus.GaugeOpts{
			Name: "active_orders",
			Help: "Number of orders currently being processed",
		},
	)

	// =============================================================================
	# 队列指标
	# =============================================================================

	// 队列长度
	queueLength = prometheus.NewGauge(
		prometheus.GaugeOpts{
			Name: "job_queue_length",
			Help: "Number of jobs in the queue",
		},
	)

	// 队列处理速率
	queueProcessingRate = prometheus.NewCounter(
		prometheus.CounterOpts{
			Name: "jobs_processed_total",
			Help: "Total number of jobs processed",
		},
	)

	// 任务等待时间
	jobWaitTime = prometheus.NewHistogram(
		prometheus.HistogramOpts{
			Name:    "job_wait_time_seconds",
			Help:    "Time jobs spend waiting in queue",
			Buckets: []float64{1, 5, 10, 30, 60, 300, 600},
		},
	)

	// =============================================================================
	# 系统指标
	# =============================================================================

	// Go 运行时指标（自动采集）
	goRuntimeInfo = prometheus.NewGaugeVec(
		prometheus.GaugeOpts{
			Name: "go_runtime_info",
			Help: "Go runtime version info",
		},
		[]string{"version"},
	)
)

func init() {
	// 注册所有指标
	// 必须在 main 函数前调用
	prometheus.MustRegister(
		httpRequestsTotal,
		httpRequestDuration,
		httpRequestsInFlight,
		ordersProcessed,
		orderProcessingDuration,
		activeOrders,
		queueLength,
		queueProcessingRate,
		jobWaitTime,
		goRuntimeInfo,
	)

	// 设置 Go 运行时信息
	goRuntimeInfo.WithLabelValues(runtime.Version()).Set(1)
}

// metricsMiddleware Prometheus 中间件
func metricsMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// 跳过指标端点本身
		if r.URL.Path == "/metrics" {
			next.ServeHTTP(w, r)
			return
		}

		// 记录正在处理的请求
		httpRequestsInFlight.Inc()
		defer httpRequestsInFlight.Dec()

		// 记录开始时间
		start := time.Now()

		// 使用自定义 ResponseWriter 包装获取状态码
		wrapped := &responseWriter{ResponseWriter: w, statusCode: http.StatusOK}
		
		// 处理请求
		next.ServeHTTP(wrapped, r)

		// 记录指标
		duration := time.Since(start).Seconds()
		status := strconv.Itoa(wrapped.statusCode)
		endpoint := normalizePath(r.URL.Path) // 规范化路径，避免路径爆炸

		httpRequestsTotal.WithLabelValues(r.Method, endpoint, status).Inc()
		httpRequestDuration.WithLabelValues(r.Method, endpoint).Observe(duration)
	})
}

// responseWriter 包装器，捕获响应状态码
type responseWriter struct {
	http.ResponseWriter
	statusCode int
}

func (rw *responseWriter) WriteHeader(code int) {
	rw.statusCode = code
	rw.ResponseWriter.WriteHeader(code)
}

// normalizePath 规范化路径，避免高基数标签
func normalizePath(path string) string {
	// 示例：将 /users/123 规范化为 /users/:id
	// 实际使用路由器提供的 PathTemplate
	return path
}

// startMetricsServer 启动指标服务器
func startMetricsServer(addr string) {
	// 指标端点
	mux := http.NewServeMux()
	mux.Handle("/metrics", promhttp.Handler())
	
	// 健康检查
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
		w.Write([]byte("OK"))
	})

	log.Printf("📊 Metrics server starting on %s", addr)
	if err := http.ListenAndServe(addr, mux); err != nil {
		log.Printf("❌ Metrics server error: %v", err)
	}
}
```

### 2.2 HPA 高级配置

#### 2.2.1 自定义指标 HPA

```yaml
# =============================================================================
# 基于 Prometheus 自定义指标的 HPA
# =============================================================================

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: user-service-hpa
  namespace: production
  labels:
    app: user-service
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: user-service

  # 副本数范围
  minReplicas: 2
  maxReplicas: 20                                     # 扩到 20 个副本

  # 扩缩容指标
  metrics:
    # =============================================================================
    # CPU 使用率
    # =============================================================================
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

    # =============================================================================
    # 内存使用率
    # =============================================================================
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80

    # =============================================================================
    # 自定义指标（需要 Prometheus Adapter）
    # =============================================================================
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second              # 自定义指标名
        target:
          type: AverageValue
          averageValue: "1000"                       # 每副本 1000 QPS

    # =============================================================================
    # 对象指标（Deployment 级别的指标）
    # =============================================================================
    - type: Object
      object:
        metric:
          name: nginx_connections_active             # Nginx 活动连接数
        describedObject:
          kind: Service
          name: user-service
          apiVersion: v1
        target:
          type: AverageValue
          averageValue: "500"                        # 平均每副本 500 连接

  # =============================================================================
  # 扩缩容行为（Kubernetes 1.27+）
  # =============================================================================
  behavior:
    # 扩容策略
    scaleUp:
      # 冷却窗口（扩容后多久开始考虑缩容）
      stabilizationWindowSeconds: 0                  # 立即考虑缩容
      
      # 扩容策略
      policies:
        # 按副本数：每 15 秒最多增加 4 个副本
        - type: Pods
          value: 4
          periodSeconds: 15
        
        # 按百分比：每 15 秒最多增加 100% 副本
        - type: Percent
          value: 100
          periodSeconds: 15
      
      # 选择最激进的策略
      selectPolicy: Max

    # 缩容策略
    scaleDown:
      # 冷却窗口（缩容前等待多久）
      stabilizationWindowSeconds: 300                # 等待 5 分钟确认稳定
      
      # 缩容策略
      policies:
        # 按副本数：每分钟最多减少 2 个副本
        - type: Pods
          value: 2
          periodSeconds: 60
        
        # 按百分比：每分钟最多减少 10%
        - type: Percent
          value: 10
          periodSeconds: 60
      
      # 选择最保守的策略
      selectPolicy: Min
```

```yaml
# =============================================================================
# Prometheus Adapter 配置
# 将 Prometheus 指标转换为 K8s Metrics API 格式
# =============================================================================

apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-adapter-config
  namespace: monitoring
data:
  config.yaml: |
    # 规则组
    rules:
      # ================================================================
      # HPA 自定义指标
      # ================================================================
      - seriesQuery: 'http_requests_total{kubernetes_pod_name!=""}'
        resources:
          overrides:
            kubernetes_pod_name:
              resource: pod
            kubernetes_namespace:
              resource: namespace
        name:
          matches: "^(.*)_total"
          as: "${1}_per_second"
        metricsQuery: sum(rate(<<.Series>>{<<.LabelMatchers>>}[2m])) by (<<.GroupBy>>)

      # 每副本 QPS
      - seriesQuery: 'http_requests_total{kubernetes_pod_name!=""}'
        resources:
          overrides:
            kubernetes_pod_name:
              resource: pod
        name:
          matches: "^(.*)_total"
          as: "${1}_per_second"
        metricsQuery: sum(rate(<<.Series>>[2m])) by (kubernetes_pod_name) / count(count(<<.Series>>) by (kubernetes_pod_name)) by (kubernetes_pod_name)

      # ================================================================
      # 业务指标
      # ================================================================
      - seriesQuery: 'active_orders{}'
        resources:
          overrides:
            namespace:
              resource: namespace
        name:
          as: "active_orders"
        metricsQuery: active_orders

      - seriesQuery: 'job_queue_length{}'
        resources:
          overrides:
            namespace:
              resource: namespace
        name:
          as: "job_queue_length"
        metricsQuery: job_queue_length

      - seriesQuery: 'jobs_processed_total{}'
        resources:
          overrides:
            namespace:
              resource: namespace
        name:
          matches: "^(.*)_total"
          as: "${1}_rate"
        metricsQuery: sum(rate(<<.Series>>[5m])) by (<<.GroupBy>>)

    # ================================================================
    # 外部指标（从 Service 获取）
    # ================================================================
    external:
      - seriesQuery: 'nginx_connections_active{}'
        resources:
          overrides:
            service:
              resource: service
        name:
          as: "nginx_connections_active"
        metricsQuery: nginx_connections_active{service="{{ $labels.service }}"}
```

```yaml
# =============================================================================
# KEDA ScaledObject
# 基于事件的弹性扩缩容
# =============================================================================

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: user-service-scaler
  namespace: production
  labels:
    app: user-service
spec:
  # =============================================================================
  # 目标 Deployment
  # =============================================================================
  scaleTargetRef:
    name: user-service

  # =============================================================================
  # 副本数范围
  # =============================================================================
  minReplicaCount: 0                                  # 缩容到零
  maxReplicaCount: 100

  # =============================================================================
  # 冷却/冷却
  # =============================================================================
  cooldownPeriod: 300                                 # 缩容冷却 5 分钟
  idleReplicaCount: 0                                # 空闲时副本数

  # =============================================================================
  # 触发器
  # =============================================================================
  triggers:
    # =============================================================================
    # Kafka 触发器
    # =============================================================================
    - type: kafka
      metadata:
        bootstrapServers: kafka.kafka.svc.cluster.local:9092
        consumerGroup: user-service-group
        topic: user-orders
        lagThreshold: "50"                            # 滞后超过 50 条消息时扩容
        offsetResetPolicy: earliest
      authenticationRef:
        name: keda-trigger-auth-kafka

    # =============================================================================
    # Redis 触发器
    # =============================================================================
    - type: redis
      metadata:
        address: redis.cache.svc.cluster.local:6379
        listName: job-queue
        listLength: "10"                              # 队列长度超过 10 时扩容
      authenticationRef:
        name: keda-trigger-auth-redis

    # =============================================================================
    # Prometheus 触发器
    # =============================================================================
    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        metricName: http_requests
        threshold: "1000"                             # QPS 超过 1000 时扩容
        query: sum(rate(http_requests_total[2m]))
      authenticationRef:
        name: keda-trigger-auth-prometheus

    # =============================================================================
    # Cron 触发器（定时扩缩容）
    # =============================================================================
    - type: cron
      metadata:
        timezone: Asia/Shanghai
        start: 0 8 * * 1-5                           # 周一至周五 8:00
        end: 0 20 * * 1-5                            # 周一至周五 20:00
        desiredReplicas: "10"                         # 扩容到 10 个副本

    # =============================================================================
    # RabbitMQ 触发器
    # =============================================================================
    - type: rabbitmq
      metadata:
        host: amqp://rabbitmq:5672
        queueName: user-tasks
        queueLength: "20"                            # 队列长度超过 20 时扩容
```

```yaml
metadata:
  name: user-service-vpa

  # =============================================================================

          memory: 64Mi

        maxAllowed:
          cpu: 2000m
          memory: 2Gi
        
        # 控制模式
        controlledResources:
          - cpu
          - memory
        
        # 建议的缓冲（推荐值 * (1 + bufferPercentage)）
        # 避免频繁调整
        controlledValues: RequestsAndLimits
```

### 3.1 扩缩容策略对比

```
┌────────────────┬─────────────────────────────────────────────────────────┐
│    指标类型     │                      适用场景                           │
├────────────────┼─────────────────────────────────────────────────────────┤
│ CPU            │ 计算密集型服务                                           │
│ Memory         │ 内存密集型服务（缓存、数据库）                           │
│ QPS            │ Web 服务、API 服务                                       │
│ 队列长度       │ 异步任务处理、消息队列消费者                             │
│ 延迟           │ SLA 敏感的服务                                           │
│ 自定义指标     │ 业务相关的指标（如订单数、在线用户）                     │
│ Cron           │ 定时任务（如大促前扩容）                                 │
└────────────────┴─────────────────────────────────────────────────────────┘
```

### 3.2 扩缩容最佳实践

```
1. 合理的副本数范围
   - 最小副本数：保证基本可用
   - 最大副本数：控制成本

2. 合理的冷却时间
   - 扩容冷却：通常不需要
   - 缩容冷却：5-15 分钟

3. 渐进式扩缩容
   - 避免频繁震荡
   - 使用 stabilizationWindow

4. 就绪检查
   - 新 Pod 必须就绪后才接收流量
   - 确保扩缩容时服务质量

5. 成本考虑
   - 缩容到零节省成本
   - 定时扩缩容应对高峰
```

### 3.3 KEDA vs HPA

```
KEDA（Event-driven Autoscaling）：
- 基于事件（队列消息数）
- 更丰富的触发器
- 缩容到零
- 适合异步任务处理

HPA（Horizontal Pod Autoscaler）：
- 基于指标（CPU、内存、自定义）
- Kubernetes 原生
- 更成熟稳定
- 适合在线服务

建议：
- 在线服务：HPA + VPA
- 异步任务：KEDA
- 组合使用
```

## 🎯 面试要点

### 问题 1：K8s 中有哪几种扩缩容方式？

**参考答案：**
```
1. HPA（Horizontal Pod Autoscaler）
   - 水平扩缩容
   - 增加/减少 Pod 副本数
   - 基于 CPU、内存、自定义指标

2. VPA（Vertical Pod Autoscaler）
   - 垂直扩缩容
   - 调整 Pod 的资源限制（CPU/内存）
   - 适合资源需求变化的应用

3. CA（Cluster Autoscaler）
   - 集群级别
   - 自动增加/减少节点
   - 基于 Pod 调度失败

4. KEDA（Kubernetes Event-driven Autoscaling）
   - 基于事件的扩缩容
   - 支持多种触发器（Kafka、Redis、Cron）
   - 适合异步任务处理

组合使用：
- HPA + VPA：在线服务
- KEDA + HPA：异步任务 + 在线服务
```

### 问题 2：HPA 的工作原理是什么？

**参考答案：**
```
HPA 工作原理：

1. 指标采集
   - Metrics API 收集指标
   - kube-controller-manager 中的 HPA Controller

2. 计算期望副本数
   desiredReplicas = ceil(currentReplicas * (currentValue / targetValue))

3. 决策
   - 期望副本数 > 当前：扩容
   - 期望副本数 < 当前：缩容

4. 执行
   - 更新 Deployment 副本数
   - Deployment Controller 创建/删除 Pod

指标来源：
- Resource Metrics（CPU/内存）：metrics-server
- Custom Metrics：prometheus-adapter
- External Metrics：其他适配器
```

### 问题 3：如何设计一个高可用的弹性架构？

**参考答案：**
```
高可用弹性架构设计：

1. 多副本部署
   - 最小副本数 >= 2
   - PodAntiAffinity 分散到不同节点

2. 健康检查
   - livenessProbe：检测进程存活
   - readinessProbe：检测是否接收流量

3. 优雅关闭
   - preStop hook
   - 等待连接关闭
   - 清理资源

4. 弹性策略
   - 合理的副本数范围
   - 冷却时间
   - 渐进式扩缩容

5. 成本优化
   - 缩容到零（非高峰期）
   - 定时扩缩容（可预测负载）
   - Spot 实例（成本敏感场景）

6. 监控告警
   - 扩容失败告警
   - 资源使用告警
   - 延迟/SLA 告警
```

### 问题 4：Go 中 GMP 调度模型是什么？

**参考答案：**
```
GMP 调度模型：

G（Goroutine）：
- 用户态轻量级协程
- 栈大小 2KB（可扩展）
- 创建成本极低

M（Machine）：
- 系统线程
- 内核态调度

P（Processor）：
- 逻辑处理器
- 执行上下文
- 本地 G 队列

工作流程：
1. M 绑定 P 获取执行权
2. 从 P 本地队列获取 G
3. G 在 M 上执行
4. G 阻塞时，M 调度其他 G

特殊场景：
- 系统调用：M 分离 P，继续调度
- 网络 IO：go netpoller 等待
- 工作窃取：空闲 P 偷其他 P 的 G

优势：
- 并发量大（百万级）
- 调度开销小（用户态）
- 资源利用率高
```

### 问题 5：如何避免扩缩容震荡？

**参考答案：**
```
扩缩容震荡原因：
- 指标波动大
- 冷却时间太短
- 阈值设置不合理

解决方案：

1. 合理设置冷却时间
   scaleDown.stabilizationWindowSeconds: 300

2. 渐进式扩缩容
   - maxSurge: 1（扩容多1个）
   - maxUnavailable: 0（缩容不少）

3. 使用平均值
   - 使用一段时间的平均值而非瞬时值
   - rate() 配合时间窗口

4. 合理阈值
   - 扩容阈值：80%
   - 缩容阈值：60%
   - 避免频繁触发

5. 最小副本数
   - 根据 SLA 保证的最小容量
   - 不能缩容到零（如果 SLA 要求）
```
