### 1.1 项目概述

这是一个基于 gRPC 的微服务订单处理系统，负责处理订单创建、支付、物流等核心业务流程。项目展示了微服务间的通信、数据一致性保障、分布式事务处理等高级特性。

**核心功能：**
- 订单创建与状态管理
- 库存预占与扣减（分布式事务）
- 订单支付对接
- 订单取消与退款
- 订单历史查询
- 订单超时自动取消

#### 2.3.1 订单处理器 handler/order.go

```go
package handler

import (
	"context"
	"fmt"
	"time"

	"order-service/internal/model"
	"order-service/internal/repository"
	"order-service/internal/service"
	pb "order-service/proto/order"

	"github.com/golang-jwt/jwt/v5"
	"gorm.io/gorm"
)

/*
Saga 模式在订单服务中的应用：

Saga 是一种分布式事务模式，将大事务拆分为多个小事务。
每个小事务都有对应的补偿事务（Compensation）。

订单创建 Saga 流程：
1. 创建订单（pending 状态）
2. 预占库存
3. 发送延迟消息（30分钟内未支付则取消）

如果某一步失败：
1. 调用前几步的补偿操作
2. 回滚订单状态
3. 释放预占的库存

补偿操作示例：
- 预占库存失败：无需补偿（还没真正扣减）
- 创建订单失败：删除订单记录
- 预占成功但后续失败：释放预占的库存
*/

type OrderHandler struct {
	pb.UnimplementedOrderServiceServer
	orderService *service.OrderService
	sagaService  *service.SagaService
}

func NewOrderHandler(orderRepo *repository.OrderRepository, sagaService *service.SagaService) *OrderHandler {
	return &OrderHandler{
		orderService: service.NewOrderService(orderRepo),
		sagaService:  sagaService,
	}
}

// CreateOrder 创建订单
func (h *OrderHandler) CreateOrder(ctx context.Context, req *pb.CreateOrderRequest) (*pb.CreateOrderResponse, error) {
	// ========== 第1步：参数验证 ==========
	if req.UserId == 0 {
		return &pb.CreateOrderResponse{
			Code:    400,
			Message: "用户ID不能为空",
			Success: false,
		}, nil
	}
	if len(req.Items) == 0 {
		return &pb.CreateOrderResponse{
			Code:    400,
			Message: "订单商品不能为空",
			Success: false,
		}, nil
	}

	// ========== 第2步：生成订单号 ==========
	// 订单号格式：ORD + 时间戳 + 随机数
	orderNo := generateOrderNo()

	// ========== 第3步：启动 Saga 事务 ==========
	// 创建 Saga 实例
	saga := h.sagaService.NewSaga(orderNo)

	// 定义 Saga 步骤
	// 每一步包含：正向操作、补偿操作
	saga.
		// 步骤1：创建订单
		Step(func(ctx context.Context) error {
			// 计算订单总金额
			var totalAmount int64
			orderItems := make([]model.OrderItem, 0, len(req.Items))

			for _, item := range req.Items {
				subtotal := item.Price * int64(item.Quantity)
				totalAmount += subtotal

				orderItems = append(orderItems, model.OrderItem{
					ProductID:   item.ProductId,
					ProductName: item.ProductName,
					Quantity:    int(item.Quantity),
					Price:       item.Price,
					Subtotal:    subtotal,
				})
			}

			// 创建订单
			order := &model.Order{
				OrderNo:     orderNo,
				UserID:      req.UserId,
				AddressID:   req.AddressId,
				Status:      model.OrderStatusPending,
				TotalAmount: totalAmount,
				ActualAmount: totalAmount,
				Remark:      req.Remark,
			}

			return saga.SetData("order", order).
				SetData("items", orderItems)
		}).
		// 步骤2：预占库存（调用库存服务）
		Step(func(ctx context.Context) error {
			// 从 Saga 上下文获取订单数据
			items, _ := saga.GetData("items").([]model.OrderItem)

			// 调用库存服务预占库存
			// 这里简化处理，实际应该 gRPC 调用库存服务
			for _, item := range items {
				// inventoryClient.ReserveStock(item.ProductID, item.Quantity)
				fmt.Printf("📦 预占库存: product=%d, quantity=%d\n", item.ProductID, item.Quantity)
			}

			return nil
		}, func(ctx context.Context) error {
			// 补偿操作：释放预占的库存
			items, _ := saga.GetData("items").([]model.OrderItem)
			for _, item := range items {
				// inventoryClient.ReleaseStock(item.ProductID, item.Quantity)
				fmt.Printf("📦 释放库存: product=%d, quantity=%d\n", item.ProductID, item.Quantity)
			}
			return nil
		}).
		// 步骤3：保存订单到数据库
		Step(func(ctx context.Context) error {
			order, _ := saga.GetData("order").(*model.Order)
			items, _ := saga.GetData("items").([]model.OrderItem)

			// 创建订单
			if err := h.orderService.Create(order, items); err != nil {
				return err
			}

			// 更新 Saga 数据
			saga.SetData("order", order)
			return nil
		}, func(ctx context.Context) error {
			// 补偿操作：删除订单
			order, _ := saga.GetData("order").(*model.Order)
			return h.orderService.Delete(order.ID)
		}).
		// 步骤4：发送延迟取消消息
		Step(func(ctx context.Context) error {
			order, _ := saga.GetData("order").(*model.Order)
			// 发送延迟消息，30分钟后检查是否支付
			// mq.SendDelayMessage("order:timeout", orderNo, 30*time.Minute)
			fmt.Printf("⏰ 订单 %s 已创建，等待支付...\n", order.OrderNo)
			return nil
		}, nil) // 最后一步不需要补偿

	// ========== 第4步：执行 Saga ==========
	order, err := saga.Execute(ctx)
	if err != nil {
		return &pb.CreateOrderResponse{
			Code:    500,
			Message: fmt.Sprintf("创建订单失败: %v", err),
			Success: false,
		}, nil
	}

	createdOrder := order.(*model.Order)

	return &pb.CreateOrderResponse{
		Code:           0,
		Message:        "订单创建成功",
		Success:        true,
		Order:          h.toProtoOrder(createdOrder),
		TransactionId:  saga.GetTransactionID(),
	}, nil
}

// GetOrder 获取订单
func (h *OrderHandler) GetOrder(ctx context.Context, req *pb.GetOrderRequest) (*pb.GetOrderResponse, error) {
	order, err := h.orderService.GetByID(req.OrderId)
	if err != nil {
		return &pb.GetOrderResponse{
			Code:    404,
			Message: "订单不存在",
			Success: false,
		}, nil
	}

	// 权限校验：只有订单所有者可以查看
	if order.UserID != req.UserId {
		return &pb.GetOrderResponse{
			Code:    403,
			Message: "无权访问此订单",
			Success: false,
		}, nil
	}

	return &pb.GetOrderResponse{
		Code:    0,
		Message: "获取成功",
		Success: true,
		Order:   h.toProtoOrder(order),
	}, nil
}

// ListOrders 订单列表
func (h *OrderHandler) ListOrders(ctx context.Context, req *pb.ListOrdersRequest) (*pb.ListOrdersResponse, error) {
	orders, total, err := h.orderService.ListByUserID(req.UserId, req.Status, req.Page, req.PageSize)
	if err != nil {
		return &pb.ListOrdersResponse{
			Code:    500,
			Message: "获取失败",
			Success: false,
		}, nil
	}

	protoOrders := make([]*pb.Order, len(orders))
	for i, order := range orders {
		protoOrders[i] = h.toProtoOrder(order)
	}

	return &pb.ListOrdersResponse{
		Code:    0,
		Message: "获取成功",
		Success: true,
		Orders:  protoOrders,
		Total:   int32(total),
	}, nil
}

// CancelOrder 取消订单
func (h *OrderHandler) CancelOrder(ctx context.Context, req *pb.CancelOrderRequest) (*pb.CancelOrderResponse, error) {
	// 获取订单
	order, err := h.orderService.GetByOrderNo(req.OrderNo)
	if err != nil {
		return &pb.CancelOrderResponse{
			Code:    404,
			Message: "订单不存在",
			Success: false,
		}, nil
	}

	// 权限校验
	if order.UserID != req.UserId {
		return &pb.CancelOrderResponse{
			Code:    403,
			Message: "无权取消此订单",
			Success: false,
		}, nil
	}

	// 状态检查
	if order.Status != model.OrderStatusPending {
		return &pb.CancelOrderResponse{
			Code:    400,
			Message: "只有待支付的订单可以取消",
			Success: false,
		}, nil
	}

	// 取消订单
	if err := h.orderService.Cancel(order.ID, req.Reason); err != nil {
		return &pb.CancelOrderResponse{
			Code:    500,
			Message: "取消失败",
			Success: false,
		}, nil
	}

	return &pb.CancelOrderResponse{
		Code:    0,
		Message: "订单已取消",
		Success: true,
	}, nil
}

// PaymentCallback 支付回调
func (h *OrderHandler) PaymentCallback(ctx context.Context, req *pb.PaymentCallbackRequest) (*pb.PaymentCallbackResponse, error) {
	// 根据订单号获取订单
	order, err := h.orderService.GetByOrderNo(req.OrderNo)
	if err != nil {
		return &pb.PaymentCallbackResponse{
			Code:    404,
			Message: "订单不存在",
		}, nil
	}

	// 幂等性检查
	// 如果订单已经是已支付状态，说明回调重复了
	if order.Status == model.OrderStatusPaid {
		return &pb.PaymentCallbackResponse{
			Code:    200,
			Message: "订单已支付",
		}, nil
	}

	// 处理支付结果
	if req.Status == 1 {
		// 支付成功
		if err := h.orderService.Paid(order.ID, req.PaymentNo); err != nil {
			return &pb.PaymentCallbackResponse{
				Code:    500,
				Message: "更新失败",
			}, nil
		}
	} else {
		// 支付失败
		if err := h.orderService.PaymentFailed(order.ID); err != nil {
			return &pb.PaymentCallbackResponse{
				Code:    500,
				Message: "更新失败",
			}, nil
		}
	}

	return &pb.PaymentCallbackResponse{
		Code:    200,
		Message: "回调处理成功",
	}, nil
}

func generateOrderNo() string {
	return fmt.Sprintf("ORD%d%s", time.Now().UnixNano(), randomString(8))

// randomString 生成随机字符串
func randomString(n int) string {
	const letters = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ"
	b := make([]byte, n)
	for i := range b {
		b[i] = letters[time.Now().UnixNano()%int64(len(letters))]
	}
	return string(b)
}
```

#### 2.3.2 Saga 事务编排 service/saga.go

```go
package service

import (
	"context"
	"fmt"
	"sync"

	"github.com/google/uuid" // UUID 生成
)

/*
Saga 事务编排器

Saga 将一个分布式事务拆分为多个步骤，每个步骤都有：
- Forward（正向操作）：执行实际的业务逻辑
- Compensate（补偿操作）：回滚之前的操作

特点：
- 最终一致性，不是强一致性
- 需要业务方实现补偿逻辑
- 适合长事务场景
*/

// SagaStep Saga 步骤
type SagaStep struct {
	ForwardFunc  func(ctx context.Context) error  // 正向操作
	CompensateFunc func(ctx context.Context) error // 补偿操作
	Description  string                              // 步骤描述（用于日志）
}

// Saga Saga 事务
type Saga struct {
	transactionID string           // 事务ID
	steps        []SagaStep        // 步骤列表
	executedSteps int              // 已执行步骤数
	data         map[string]interface{} // 事务数据
	mu           sync.RWMutex
	err          error             // 事务错误
}

// NewSaga 创建新的 Saga
func NewSaga(transactionID string) *Saga {
	return &Saga{
		transactionID: transactionID,
		steps:        make([]SagaStep, 0),
		data:         make(map[string]interface{}),
	}
}

// GetTransactionID 获取事务ID
func (s *Saga) GetTransactionID() string {
	return s.transactionID
}

// Step 添加步骤
// forward: 正向操作
// compensate: 补偿操作（可选）
func (s *Saga) Step(forward func(ctx context.Context) error, compensate func(ctx context.Context) error) *Saga {
	s.steps = append(s.steps, SagaStep{
		ForwardFunc:     forward,
		CompensateFunc:  compensate,
		Description:     fmt.Sprintf("Step %d", len(s.steps)+1),
	})
	return s
}

// SetData 设置事务数据
func (s *Saga) SetData(key string, value interface{}) *Saga {
	s.mu.Lock()
	defer s.mu.Unlock()
	s.data[key] = value
	return s
}

// GetData 获取事务数据
func (s *Saga) GetData(key string) interface{} {
	s.mu.RLock()
	defer s.mu.RUnlock()
	return s.data[key]
}

// Execute 执行 Saga
func (s *Saga) Execute(ctx context.Context) (interface{}, error) {
	fmt.Printf("🚀 开始执行 Saga: %s\n", s.transactionID)

	// 按顺序执行所有步骤
	for i, step := range s.steps {
		s.executedSteps = i + 1
		fmt.Printf("📍 执行步骤 %d/%d: %s\n", i+1, len(s.steps), step.Description)

		// 执行正向操作
		if err := step.ForwardFunc(ctx); err != nil {
			fmt.Printf("❌ 步骤 %d 失败: %v\n", i+1, err)
			s.err = err

			// 执行补偿操作（从后往前回滚）
			s.compensate(ctx)
			return nil, fmt.Errorf("Saga 执行失败 at step %d: %w", i+1, err)
		}

		fmt.Printf("✅ 步骤 %d 完成\n", i+1)
	}

	fmt.Printf("✅ Saga 执行成功: %s\n", s.transactionID)

	// 返回设置的数据
	return s.GetData("order"), nil
}

// compensate 执行补偿操作
func (s *Saga) compensate(ctx context.Context) {
	fmt.Printf("🔄 开始补偿操作，已执行 %d 步\n", s.executedSteps-1)

	// 从后往前执行补偿
	for i := s.executedSteps - 2; i >= 0; i-- {
		step := s.steps[i]

		if step.CompensateFunc == nil {
			fmt.Printf("⏭️  步骤 %d 无补偿操作，跳过\n", i+1)
			continue
		}

		fmt.Printf("🔄 执行补偿: %s\n", step.Description)
		if err := step.CompensateFunc(ctx); err != nil {
			// 补偿失败，记录错误但继续执行其他补偿
			// 实际生产中应该记录日志并人工介入
			fmt.Printf("⚠️  补偿失败 at step %d: %v\n", i+1, err)
		} else {
			fmt.Printf("✅ 补偿成功: %s\n", step.Description)
		}
	}
}
```

#### 2.3.3 分布式锁 redis/lock.go

```go
package redis

import (
	"context"
	"errors"
	"time"

	"github.com/redis/go-redis/v9"
)

/*
Redis 分布式锁

分布式锁用于在多进程/多服务器环境下保证互斥访问共享资源。

实现原理：
1. 使用 SET key value NX PX timeout 命令尝试获取锁
   - NX: 只有 key 不存在时才设置
   - PX: 设置过期时间（毫秒）
2. 释放锁时验证 value 是否匹配（防止误释放他人持有的锁）
3. 使用 Lua 脚本保证释放锁的原子性

特点：
- 互斥性：同一时刻只有一个持有者
- 超时释放：防止持有者崩溃导致死锁
- 非阻塞：获取失败立即返回
*/

var (
	// ErrLockNotAcquired 获取锁失败
	ErrLockNotAcquired = errors.New("获取锁失败")

	// ErrLockNotHeld 锁未被持有
	ErrLockNotHeld = errors.New("锁未被持有")
)

// Lock Redis 分布式锁
type Lock struct {
	client *redis.Client
	key    string
	value  string
	ttl    time.Duration
}

// NewLock 创建锁实例
// key: 锁的键名
// value: 锁的值（通常使用唯一ID，如 UUID）
// ttl: 锁的过期时间
func NewLock(client *redis.Client, key, value string, ttl time.Duration) *Lock {
	return &Lock{
		client: client,
		key:    key,
		value:  value,
		ttl:    ttl,
	}
}

// Acquire 尝试获取锁
// ctx: 上下文，用于控制超时和取消
// retryTimes: 重试次数
// retryDelay: 重试间隔
//
// 返回：
// - 成功获取锁：true
// - 获取失败：false
func (l *Lock) Acquire(ctx context.Context, retryTimes int, retryDelay time.Duration) (bool, error) {
	// 循环尝试获取锁
	for i := 0; i <= retryTimes; i++ {
		// SET key value NX PX timeout
		// NX: 只有 key 不存在时才设置（保证互斥）
		// PX: 设置过期时间（防止死锁）
		ok, err := l.client.SetNX(ctx, l.key, l.value, l.ttl).Result()
		if err != nil {
			return false, err
		}

		if ok {
			return true, nil
		}

		// 获取失败，等待后重试
		if i < retryTimes {
			select {
			case <-ctx.Done():
				return false, ctx.Err()
			case <-time.After(retryDelay):
				// 继续重试
			}
		}
	}

	return false, ErrLockNotAcquired
}

// Release 释放锁
// 只有锁的持有者才能释放锁
// 使用 Lua 脚本保证原子性
func (l *Lock) Release(ctx context.Context) error {
	// Lua 脚本：只有 value 匹配时才删除
	// 这样可以确保只删除自己持有的锁
	// 防止误删他人持有的锁
	script := redis.NewScript(`
		if redis.call("get", KEYS[1]) == ARGV[1] then
			return redis.call("del", KEYS[1])
		else
			return 0
		end
	`)

	result, err := script.Run(ctx, l.client, []string{l.key}, l.value).Result()
	if err != nil {
		return err
	}

	if result.(int64) == 0 {
		return ErrLockNotHeld
	}

	return nil
}

// Extend 延长锁的持有时间
// 如果业务处理时间超过预期，需要续期
func (l *Lock) Extend(ctx context.Context, ttl time.Duration) error {
	// Lua 脚本：只有 value 匹配时才延长过期时间
	script := redis.NewScript(`
		if redis.call("get", KEYS[1]) == ARGV[1] then
			return redis.call("pexpire", KEYS[1], ARGV[2])
		else
			return 0
		end
	`)

	result, err := script.Run(ctx, l.client, []string{l.key}, l.value, ttl.Milliseconds()).Result()
	if err != nil {
		return err
	}

	if result.(int64) == 0 {
		return ErrLockNotHeld
	}

	return nil
}

// Client 封装 Redis 客户端，提供分布式锁功能
type Client struct {
	client *redis.Client
}

// NewClient 创建客户端
func NewClient(addr, password string, db int) *Client {
	return &Client{
		client: redis.NewClient(&redis.Options{
			Addr:     addr,
			Password: password,
			DB:       db,
		}),
	}
}

// NewLock 创建一个新的分布式锁
func (c *Client) NewLock(key string, ttl time.Duration) *Lock {
	// 使用 UUID 作为锁的值，确保唯一性
	value := generateUUID()
	return NewLock(c.client, key, value, ttl)
}

// generateUUID 生成 UUID
func generateUUID() string {
	return time.Now().Format("20060102150405.0000000000")
}
```

## 🎯 面试要点

### 问题 1：gRPC 和 RESTful API 的区别是什么？

**参考答案：**
```
通信协议：
- RESTful: HTTP/1.1 或 HTTP/2
- gRPC: HTTP/2

数据格式：
- RESTful: JSON（人类可读）
- gRPC: Protobuf（二进制，效率高）

接口定义：
- RESTful: 无标准定义（OpenAPI/Swagger 是后来补充）
- gRPC: .proto 文件（强类型）

通信模式：
- RESTful: 请求-响应
- gRPC: 支持四种模式（Unary、Client Streaming、Server Streaming、Bidirectional Streaming）

代码生成：
- RESTful: 需要手动编写客户端
- gRPC: 自动生成多语言客户端

适用场景：
- RESTful: 浏览器客户端、公开 API
- gRPC: 微服务内部通信、移动端
```

### 问题 2：什么是 Protobuf？相比 JSON 有什么优势？

**参考答案：**
```
Protobuf（Protocol Buffers）：
- Google 开源的序列化协议
- 使用 .proto 文件定义数据结构
- 自动生成多语言代码

优势：
1. 体积小：二进制格式，比 JSON 小 3-10 倍
2. 速度快：解析速度比 JSON 快 10-100 倍
3. 类型安全：编译时检查类型
4. 向后兼容：可以添加新字段而不破坏旧代码
5. 代码生成：自动生成序列化代码

劣势：
1. 二进制格式，人类不可读
2. 需要定义 .proto 文件
3. 学习成本
```

### 问题 3：Saga 模式和 TCC 模式有什么区别？

**参考答案：**
```
Saga 模式：
- 每个步骤都有正向操作和补偿操作
- 补偿是"撤销"操作，不是"逆操作"
- 简单易实现
- 适合：事务结果可补偿的业务

TCC 模式（Try-Confirm-Cancel）：
- Try：预留资源（锁定）
- Confirm：确认操作（真正执行）
- Cancel：取消操作（释放资源）
- 更强的一致性保证
- 实现复杂，需要幂等性支持
- 适合：对一致性要求高的业务

选择建议：
- 业务可补偿：优先 Saga
- 需要强一致性：考虑 TCC
- 简单场景：使用本地消息表 + 定时任务
```

### 问题 4：Go 中 GMP 调度模型是什么？

**参考答案：**
```
GMP 模型：
- G（Goroutine）：协程，用户态轻量级线程
- M（Machine）：系统线程，对应 OS 线程
- P（Processor）：处理器，运行 G 的上下文

工作流程：
1. 线程 M 从 P 的本地队列获取 G
2. G 执行系统调用时，M 会绑定的 P 分离
3. 空闲的 P 会从全局队列或其他 P 偷取 G

特点：
- 轻量级：创建成本低（2KB vs 1MB）
- 调度在用户态完成，无需内核介入
- 支持数十万并发
- 自动调度，开发者无需关心

调度策略：
- 公平调度：轮转执行
- 工作窃取：当 P 空闲时，从其他 P 偷取 G
- 系统调用：G 执行系统调用时，不阻塞 M
```

### 问题 5：如何保证分布式系统中的幂等性？

**参考答案：**
```
幂等性：多次执行产生相同结果

实现方式：
1. 数据库唯一索引
   - 对关键字段建立唯一索引
   - 重复插入会报错

2. 去重表
   - 记录已处理的请求 ID
   - 处理前先查询

3. 状态机
   - 订单状态只能正向流转
   - 重复回调不会重复处理

4. 分布式锁 + 业务标记
   - 使用唯一键加锁
   - 标记处理状态

实际应用：
- 支付回调：先查状态，再更新
- 订单创建：使用幂等键
- 库存扣减：乐观锁版本号
```
